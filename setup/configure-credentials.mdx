GX Core needs credentials to connect to your data sources. Because credentials provide access to your data, they must never be hardcoded in scripts or committed to version control.

GX Core supports two secure storage methods:

- **Environment variables** — Available immediately in any session; easy to set in CI/CD.
- **`config_variables.yml`** — A YAML file in the `uncommitted/` directory of a File Data Context; useful when managing many variables or different environments.

Both methods use string substitution: you reference a variable name in your connection string using `${VAR_NAME}`, and GX replaces it with the stored value at runtime.

<Warning>
Never hardcode passwords, tokens, or connection strings directly in your Python scripts or configuration files that are tracked by version control. Always use environment variables or `config_variables.yml`.
</Warning>

## Store credentials

<Tabs>
  <Tab title="Environment variables">
    Environment variables are the fastest way to configure credentials. Export them in your terminal session or add them to your shell profile (`~/.bashrc` or `~/.zshrc`) for persistence across sessions.

    **Export individual credentials:**

    ```bash
    export MY_POSTGRES_USERNAME=myuser
    export MY_POSTGRES_PASSWORD=mypassword
    ```

    **Or export a full connection string:**

    ```bash
    export POSTGRES_CONNECTION_STRING="postgresql+psycopg2://myuser:mypassword@localhost:5432/mydb"
    ```

    **Or combine both — reference individual variables inside a connection string variable:**

    ```bash
    export MY_POSTGRES_USERNAME=myuser
    export MY_POSTGRES_PASSWORD=mypassword
    export POSTGRES_CONNECTION_STRING="postgresql+psycopg2://${MY_POSTGRES_USERNAME}:${MY_POSTGRES_PASSWORD}@localhost:5432/mydb"
    ```

    <Note>
    The `$` character signals string substitution. If your password contains a literal `$`, escape it with a backslash. For example, if your password is `pa$$word`:

    ```bash
    export MY_POSTGRES_PASSWORD=pa\$\$word
    ```
    </Note>

    Environment variables set with `export` in a terminal session do not persist after the session ends. Add them to `~/.bashrc` or `~/.zshrc` to make them permanent.
  </Tab>

  <Tab title="config_variables.yml">
    A File Data Context stores a `config_variables.yml` file at `great_expectations/uncommitted/config_variables.yml`. The `uncommitted/` directory is included in the default `.gitignore`, so its contents are excluded from version control.

    <Info>
    You must have a [File Data Context](/setup/data-context) before you can use `config_variables.yml`. Create one with `gx.get_context(mode="file")`.
    </Info>

    **Store individual credentials:**

    ```yaml
    MY_POSTGRES_USERNAME: myuser
    MY_POSTGRES_PASSWORD: mypassword
    ```

    **Or store a full connection string:**

    ```yaml
    POSTGRES_CONNECTION_STRING: "postgresql+psycopg2://myuser:mypassword@localhost:5432/mydb"
    ```

    **Or reference individual variables inside a connection string variable:**

    ```yaml
    MY_POSTGRES_USERNAME: myuser
    MY_POSTGRES_PASSWORD: mypassword
    POSTGRES_CONNECTION_STRING: "postgresql+psycopg2://${MY_POSTGRES_USERNAME}:${MY_POSTGRES_PASSWORD}@localhost:5432/mydb"
    ```

    <Note>
    If your password contains a literal `$`, escape it with a backslash: `pa\$\$word`.
    </Note>

    YAML config variables provide a cleaner interface for managing many credentials. You can also create separate files for different environments (development, staging, production) and swap them as needed.
  </Tab>
</Tabs>

## Reference credentials in Python

Once credentials are stored, reference them in your Python code using `${VAR_NAME}` syntax inside any string that GX Core accepts as a connection parameter. GX substitutes the variable at runtime.

**Reference individual variables in a connection string:**

```python
connection_string = "postgresql+psycopg2://${MY_POSTGRES_USERNAME}:${MY_POSTGRES_PASSWORD}@localhost:5432/mydb"
```

**Or reference a variable that holds the full connection string:**

```python
connection_string = "${POSTGRES_CONNECTION_STRING}"
```

GX Core performs the substitution automatically when the string is passed to a method that uses it, such as when creating a Data Source.

## Examples by data source

<AccordionGroup>
  <Accordion title="PostgreSQL">
    **1. Store credentials:**

    ```bash
    export PG_USERNAME=myuser
    export PG_PASSWORD=mypassword
    export PG_HOST=localhost
    export PG_PORT=5432
    export PG_DATABASE=mydb
    ```

    **2. Reference in Python:**

    ```python
    import great_expectations as gx

    context = gx.get_context()

    data_source = context.data_sources.add_postgres(
        name="my_postgres",
        connection_string="postgresql+psycopg2://${PG_USERNAME}:${PG_PASSWORD}@${PG_HOST}:${PG_PORT}/${PG_DATABASE}",
    )
    ```

    The PostgreSQL connection string format is:

    ```
    postgresql+psycopg2://<USER_NAME>:<PASSWORD>@<HOST>:<PORT>/<DATABASE>
    ```

    Install the PostgreSQL extra if you haven't already:

    ```bash
    pip install 'great_expectations[postgresql]'
    ```
  </Accordion>

  <Accordion title="Snowflake">
    Snowflake uses separate named parameters rather than a single connection string. Snowflake has deprecated password authentication — use key-pair authentication for new Data Sources.

    **1. Store credentials:**

    ```bash
    export SNOWFLAKE_ACCOUNT=myaccount
    export SNOWFLAKE_USER=myuser
    export SNOWFLAKE_DATABASE=mydb
    export SNOWFLAKE_SCHEMA=myschema
    export SNOWFLAKE_WAREHOUSE=mywarehouse
    export SNOWFLAKE_ROLE=myrole
    export SNOWFLAKE_PRIVATE_KEY=<unencrypted_private_key_contents>
    ```

    **2. Reference in Python:**

    ```python
    import great_expectations as gx

    context = gx.get_context()

    data_source = context.data_sources.add_snowflake(
        name="my_snowflake",
        account="${SNOWFLAKE_ACCOUNT}",
        user="${SNOWFLAKE_USER}",
        database="${SNOWFLAKE_DATABASE}",
        schema="${SNOWFLAKE_SCHEMA}",
        warehouse="${SNOWFLAKE_WAREHOUSE}",
        role="${SNOWFLAKE_ROLE}",
        private_key="${SNOWFLAKE_PRIVATE_KEY}",
    )
    ```

    <Warning>
    Do not include the `-----BEGIN PRIVATE KEY-----` / `-----END PRIVATE KEY-----` markers in the `private_key` value. Use an unencrypted key.
    </Warning>

    Install the Snowflake extra if you haven't already:

    ```bash
    pip install 'great_expectations[snowflake]'
    ```
  </Accordion>

  <Accordion title="Amazon S3">
    GX Core uses `boto3` to access Amazon S3. Credentials are managed through the AWS CLI rather than GX config variables.

    **1. Configure AWS credentials:**

    ```bash
    aws configure
    ```

    This stores your `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and default region in `~/.aws/credentials`.

    Alternatively, export them as environment variables:

    ```bash
    export AWS_ACCESS_KEY_ID=your_key_id
    export AWS_SECRET_ACCESS_KEY=your_secret_key
    export AWS_DEFAULT_REGION=us-east-1
    ```

    **2. Verify your credentials:**

    ```bash
    aws sts get-caller-identity
    ```

    When credentials are correctly configured, your `UserId`, `Account`, and `Arn` are returned.

    **3. Use S3 in Python:**

    ```python
    import great_expectations as gx

    context = gx.get_context()

    data_source = context.data_sources.add_pandas_s3(
        name="my_s3_source",
        bucket="my-bucket",
    )
    ```

    Install the S3 extra if you haven't already:

    ```bash
    pip install 'great_expectations[s3]'
    ```
  </Accordion>
</AccordionGroup>

## Connection string reference

The following connection string formats are accepted by GX Core for SQL-based Data Sources:

| Database | Connection string format |
|---|---|
| PostgreSQL | `postgresql+psycopg2://<USER_NAME>:<PASSWORD>@<HOST>:<PORT>/<DATABASE>` |
| Redshift | `redshift+psycopg2://<USER_NAME>:<PASSWORD>@<HOST>:<PORT>/<DATABASE>?sslmode=<SSLMODE>` |
| Databricks SQL | `databricks://token:<TOKEN>@<HOST>:<PORT>?http_path=<HTTP_PATH>&catalog=<CATALOG>&schema=<SCHEMA>` |
| BigQuery | `bigquery://<GCP_PROJECT>/<BIGQUERY_DATASET>?credentials_path=/path/to/credentials.json` |
| SQLite | `sqlite:///<PATH_TO_DB_FILE>` |

Microsoft Fabric, Microsoft SQL Server, and Snowflake use separate named parameters instead of a single connection string. Refer to the GX Core API documentation for those parameter names.

## Using secrets managers

GX Core also supports pulling credentials from secrets management services. See the GX Core documentation on secrets managers for setup instructions:

- AWS Secrets Manager — `pip install 'great_expectations[aws_secrets]'`
- Google Cloud Secret Manager
- Azure Key Vault

Values in `config_variables.yml` that begin with `secret|arn:aws:secretsmanager` are substituted with the corresponding value from AWS Secrets Manager at runtime.
